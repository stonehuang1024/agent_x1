# gstack 模块与关键文件速查表

## 这份文档怎么用

前面几篇文档更偏“讲道理”和“讲系统”。

这篇文档更偏“查表”和“导航”。

适合这些场景：

- 你已经理解 `gstack` 的大方向，现在要快速找到源码入口。
- 你准备修改某个模块，但不想先通读整个仓库。
- 你想知道某个命令、某个状态文件、某个脚本到底归谁管。
- 你想把前面几篇长文档里的概念，精确落到文件路径上。

## 模块之间最容易被忽视的依赖关系

| 上游 | 下游 | 容易忽视的点 |
|---|---|---|
| `commands.ts` | `gen-skill-docs.ts` | 命令文档不是手写的，而是从注册表反向生成 |
| `snapshot.ts` | skills 文档与 parser | flags 是单一事实源，文档和校验都依赖它 |
| `setup` | `.agents/skills` / 宿主可见技能 | skills 需要生成，不是 clone 完就天然有效 |
| `office-hours` 产物 | `plan-*` | 计划阶段高度依赖上游设计文档质量 |
| `feedback-pending.json` | design 迭代 agent | 设计 board 与主 agent 间的反馈桥梁不是 stdout，而是文件协议 |
| `learnings.jsonl` | `review` / `qa` / `benchmark` | learnings 会影响后续技能的建议密度和方向 |
| `browse.json` | CLI / extension / sidebar-agent | 一个 state file 串起了多个参与者 |
| worktree harvest | E2E / 子会话实验 | patch 收割是长期演化和验证的重要资产 |

## 这份速查表的最后结论

如果说前面四篇文档解决的是：

- 它是什么
- 为什么这么设计
- 为什么需要这么多 agent
- 能否长期运行

那么这篇速查表解决的是：

**“我现在要下手读源码或改东西，应该先看哪里？”**

对 `gstack` 这种多层系统来说，这个问题非常关键。

因为它的复杂不在某个 2000 行文件里，而在于：

- 角色层
- 运行时层
- 生成层
- 文件状态层
- 观测层
- 云端最小服务层

是一起协作的。

而这份文件的目标，就是把这种“系统级复杂性”拆回一张可以快速导航的地图。
